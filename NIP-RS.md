NIP-RS
======

Cross-Device Read State Sync
-----------------------------

`draft` `optional`

## Abstract

This NIP defines a scheme for synchronizing per-context read state (e.g., "read up to timestamp T") across multiple client instances belonging to the same user, using encrypted `kind:30078` events.

## Motivation

A user running Nostr clients on multiple devices (phone, desktop, web) has no way to share read position across those clients. Each instance independently tracks what has been read, causing already-read content to appear unread on other devices.

This NIP defines a minimal, privacy-preserving protocol for propagating read state across client instances without requiring relay-side logic or coordination between different client implementations.

## Specification

### Event Structure

Clients publish a `kind:30078` addressable event (per [NIP-78](78.md)) with the following structure:

```jsonc
{
  "kind": 30078,
  "pubkey": "<user-pubkey>",
  "created_at": <unix-timestamp>,
  "tags": [
    ["d", "read-state:<client-id>"]
  ],
  "content": "<nip44-encrypted-json>"
}
```

Clients MAY include a `["t", "read-state"]` tag to enable relay-side filtering. This slightly increases metadata visibility (the tag is queryable) but reduces fetch overhead for users with many `kind:30078` events.

#### `d` Tag

The `d` tag MUST be `read-state:<client-id>`, where `<client-id>` is a random opaque string (e.g., 16 hex bytes) generated by the client on first launch and persisted locally. Each client instance MUST use a stable, unique `client-id` for the lifetime of that installation.

#### Content

The `content` field MUST be a [NIP-44](44.md) ciphertext. The NIP-44 conversation key MUST be computed as `nip44_conversation_key(user_privkey, user_pubkey)` — the user's private key as the local party and their own public key as the remote party.

The plaintext MUST be a JSON object of the following form:

```json
{
  "v": 1,
  "contexts": {
    "<context-id>": <unix-timestamp>
  }
}
```

- `v` is an integer schema version. Clients MUST ignore blobs with unknown `v` values. Future schema versions will be defined in amendments to this NIP. No migration path is defined for `v:1`.
- Keys under `contexts` are arbitrary UTF-8 strings identifying a readable context (e.g., a channel, group, or conversation). The format shown in examples is illustrative only; this NIP does not prescribe context identifier format.
- Values are unix timestamps (integer seconds) representing "all messages in this context at or before this time have been read."

#### Context Identifiers

Context identifier format is not prescribed by this NIP. Clients choose identifiers appropriate to their context type (e.g., a NIP-28 channel event ID, a NIP-29 group address, a pubkey for DMs). Interoperability between different client implementations on context ID conventions is outside the scope of this NIP.

### Fetching

To load read state, a client MUST fetch all `kind:30078` events for the user within the time horizon and filter locally for `d` tags beginning with `read-state:`. If the `t` tag was published, clients MAY use the more targeted filter:

```json
{"kinds": [30078], "authors": ["<user-pubkey>"], "#t": ["read-state"], "since": <now - horizon>}
```

If the `t` tag was not published, clients MUST fall back to fetching all `kind:30078` events and filtering locally by `d` tag prefix:

```json
{"kinds": [30078], "authors": ["<user-pubkey>"], "since": <now - horizon>}
```

Clients SHOULD limit the fetch to events with `created_at` within a configurable time horizon (default: 7 days). Absence of a context in all fetched blobs means the read state for that context is **unknown** — clients SHOULD treat unknown contexts as unread (conservative default). The horizon is a storage and fetch optimization, not a semantic claim about read status. Contexts that were read but have aged out of the time horizon are indistinguishable from never-read contexts. Clients MAY extend the horizon or maintain a local cache to mitigate this.

### Merge Rule

After decrypting all fetched blobs, the effective read timestamp for each context is:

```
effective[context] = max(timestamp) across all blobs
```

This is a grow-only max-register state-based CvRDT with an associative, commutative, idempotent join. Clients MUST NOT lower a read timestamp — only advance it.

### Writing

Clients MUST NOT publish read state without explicit user opt-in. Opt-in is a persistent user preference, not enabled by default.

Each client instance maintains its own blob (one `kind:30078` event per `client-id`). Writing replaces the previous blob via parameterized replaceable event semantics ([NIP-33](33.md)).

#### Read-Before-Write

Before publishing, a client MUST:

1. Fetch its own current blob from each relay it intends to publish to, and merge all fetched versions.
2. Decrypt and merge the fetched blob with local state using `max()` per context.
3. Publish the merged result.

The `created_at` monotonicity rule applies relative to the maximum `created_at` seen across all fetched blobs. This ensures no state is lost when two instances write concurrently.

#### Live Subscription and Convergence

Clients SHOULD maintain a live subscription to their own `kind:30078` events. When a blob from another client instance arrives:

1. Merge it into local state using `max()` per context.
2. If any context timestamp in the incoming blob is greater than the corresponding timestamp in the client's last-published blob (or the context is absent from the last-published blob), re-publish the client's own blob after a debounce delay.
3. Clients MUST suppress the re-publish if the merged result is identical to the client's last-published blob. A client that has never published treats its last-published blob as empty.
4. Clients SHOULD limit re-publishes triggered by incoming blobs to at most one per debounce window, regardless of how many blobs arrive during that window.

This drives convergence without a coordination round-trip.

#### Clock Skew

When publishing, if the client's local clock produces a `created_at` value less than or equal to the maximum `created_at` seen across all fetched blobs for the same `d` tag, the client MUST use `max_fetched_created_at + 1` instead.

#### Debounce and Pruning

Clients SHOULD debounce writes to avoid excessive relay traffic (e.g., flush 5–10 seconds after the last local read-state change, or on app close/background transition). Clients MUST NOT write on every individual read action.

The blob SHOULD contain only contexts the client has explicitly interacted with. Clients SHOULD prune aggressively, prioritizing recently-active contexts, and MAY drop entries older than the time horizon before writing. Clients MUST ensure the published event does not exceed relay event size limits (typically 64 KB content).

## Example

A user runs two clients: a desktop app (`client-id: a1b2c3d4`) and a mobile app (`client-id: e5f6a7b8`).

Desktop blob (`d` tag: `read-state:a1b2c3d4`), decrypted content:
```json
{
  "v": 1,
  "contexts": {
    "ctx:AAA": 1700000100,
    "ctx:BBB": 1700000050
  }
}
```

Mobile blob (`d` tag: `read-state:e5f6a7b8`), decrypted content:
```json
{
  "v": 1,
  "contexts": {
    "ctx:AAA": 1700000200,
    "ctx:CCC": 1700000080
  }
}
```

Merged effective state:
```json
{
  "ctx:AAA": 1700000200,
  "ctx:BBB": 1700000050,
  "ctx:CCC": 1700000080
}
```

## Privacy Considerations

The `content` field is NIP-44 encrypted to the user's own keypair. Context identifiers and timestamps are not visible to relay operators or other users. As with all NIP-44 encrypt-to-self data, compromise of the user's private key exposes all stored read state.

The `d` tag prefix `read-state:` and the number of distinct `client-id` values are visible to relay operators, revealing that the user employs read-state sync and approximately how many client instances they run. Write frequency may reveal approximate activity level.

The stable `client-id` is a persistent pseudonymous device identifier visible to relay operators over time. Users MAY periodically rotate their `client-id`, accepting that blobs published under the old identifier become orphaned and are eventually ignored as they age out of the time horizon. Rotation mechanics (e.g., tombstoning the old blob) are left to client discretion.

Because the merge rule is monotonic, replaying an old event to a relay is harmless — it cannot lower a read timestamp.

Clients supporting multiple Nostr identities SHOULD use distinct `client-id` values per identity. Reusing a `client-id` across pubkeys allows relay operators to link those identities.

This feature is opt-in. Clients MUST NOT publish read state events unless the user has explicitly enabled the feature.

## Backwards Compatibility

This NIP introduces no changes to existing event kinds or relay behavior. It uses only standard NIP-01 event storage, NIP-33 addressable event semantics, NIP-44 encryption, and NIP-78 application data conventions. Clients that do not implement this NIP are unaffected.
