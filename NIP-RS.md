NIP-RS
======

Cross-Device Read State Sync
-----------------------------

`draft` `optional`

## Abstract

This NIP defines a scheme for synchronizing per-context read state (e.g., "read up to timestamp T") across multiple client instances belonging to the same user, using encrypted `kind:30078` events.

## Motivation

A user running Nostr clients on multiple devices (phone, desktop, web) has no way to share read position across those clients. Each instance independently tracks what has been read, causing already-read content to appear unread on other devices.

This NIP defines a minimal, privacy-preserving protocol for propagating read state across client instances without requiring relay-side logic or coordination between different client implementations.

## Non-Goals

This NIP intentionally does not address:

- **Durable read log.** Read state is best-effort recent activity hints. Contexts that age out of the time horizon are indistinguishable from never-read contexts.
- **Cross-client interoperability.** Context identifier format is not prescribed. Different client families are not expected to share read state.
- **Mark-as-unread.** The merge rule is monotonic (`max()`). Once a context is marked read up to time T, it cannot be moved backward.
- **Guaranteed ordering.** Late-arriving messages with old timestamps may appear as already-read. This is a consequence of timestamp-based tracking and is accepted.

## Specification

### Event Structure

Clients publish a `kind:30078` addressable event (per [NIP-78](78.md)) with the following structure:

```jsonc
{
  "kind": 30078,
  "pubkey": "<user-pubkey>",
  "created_at": <unix-timestamp>,
  "tags": [
    ["d", "read-state:<slot-id>"],
    ["t", "read-state"]
  ],
  "content": "<nip44-encrypted-json>"
}
```

#### `d` Tag

The `d` tag MUST be `read-state:<slot-id>`, where `<slot-id>` is a random opaque string (e.g., 16 random hex bytes) generated by the client on first launch and persisted locally. The slot ID is used solely for addressability — it MUST NOT be the client's `client_id` or derived from it. Each client instance MUST use a stable, unique slot ID for the lifetime of that installation.

#### `t` Tag

Clients MUST include a `["t", "read-state"]` tag. This enables relay-side filtering without requiring clients to fetch all `kind:30078` events.

#### Content

The `content` field MUST be a [NIP-44](44.md) ciphertext. The NIP-44 conversation key MUST be computed as `nip44_conversation_key(user_privkey, user_pubkey)` — the user's private key as the local party and their own public key as the remote party.

The plaintext MUST be a JSON object of the following form:

```json
{
  "v": 1,
  "client_id": "<client-id>",
  "contexts": {
    "<context-id>": <unix-timestamp>
  }
}
```

- `v` — integer schema version. Clients MUST ignore blobs with unknown `v` values. Future schema versions will be defined in amendments to this NIP.
- `client_id` — a random opaque string (e.g., 16 hex bytes) identifying this client instance. Generated on first launch and persisted locally. Clients SHOULD preserve the same `client_id` for as long as possible (across updates, restarts, etc.) to avoid creating duplicate blobs.
- `contexts` — keys are arbitrary UTF-8 strings identifying a readable context (e.g., a channel, group, or conversation). Values are unix timestamps (integer seconds) representing "all messages in this context at or before this time have been read."

Clients SHOULD use consistent, accurate timestamps derived from message ordering or relay-provided timestamps rather than local wall-clock time. Inaccurate timestamps degrade the quality of read state for the publishing client but do not affect other users.

#### Context Identifiers

Context identifier format is not prescribed by this NIP. Clients choose identifiers appropriate to their context type (e.g., a NIP-28 channel event ID, a NIP-29 group address, a pubkey for DMs). Interoperability between different client implementations on context ID conventions is outside the scope of this NIP.

### Validation and Error Handling

Clients MUST discard events that fail validation. Specifically:

- Events with zero `d` tags, multiple `d` tags, or a `d` tag value not matching the prefix `read-state:` MUST be ignored.
- Events whose decrypted content is not valid JSON, or whose JSON does not conform to the schema above (missing `v`, missing `contexts`, non-integer timestamps, non-object `contexts`) MUST be ignored.
- Unknown keys in the decrypted JSON SHOULD be ignored (forward compatibility).
- Blobs with an unknown `v` value MUST be ignored entirely.

Clients SHOULD enforce reasonable bounds on blob contents:

- Context count SHOULD NOT exceed 10,000 entries per blob.
- Context identifier length SHOULD NOT exceed 256 bytes.
- Timestamps SHOULD be positive integers within a plausible range (e.g., after 2020-01-01 and not more than 1 hour in the future).

These limits are defensive guidance, not consensus rules. Clients MAY choose stricter or looser bounds appropriate to their implementation.

### Fetching

To load read state, a client MUST fetch all `kind:30078` events for the user using the `t` tag filter:

```json
{"kinds": [30078], "authors": ["<user-pubkey>"], "#t": ["read-state"], "since": <now - horizon>}
```

Clients SHOULD limit the fetch to events with `created_at` within a configurable time horizon (default: 7 days). The horizon is a storage and fetch optimization, not a semantic claim about read status.

After fetching, the client MUST decrypt each blob and inspect the `client_id` field to identify its own blob. Blobs whose `client_id` matches the client's own are used for read-before-write; all blobs (including the client's own) are merged to compute effective read state.

Absence of a context in all fetched blobs means the read state for that context is **unknown** — clients SHOULD treat unknown contexts as unread (conservative default). Clients MAY extend the horizon or maintain a local cache to mitigate loss of state at the horizon boundary.

### Merge Rule

After decrypting all fetched blobs from all client instances, the effective read timestamp for each context is:

```
effective[context] = max(timestamp) across all blobs
```

This is a grow-only max-register state-based CvRDT with an associative, commutative, idempotent join. Clients MUST NOT lower a read timestamp — only advance it.

### Writing

Clients MUST NOT publish read state without explicit user opt-in. Opt-in is a persistent user preference, not enabled by default.

Each client instance maintains its own blob (one `kind:30078` event per slot ID). A client MUST only update the blob whose decrypted `client_id` matches its own. Writing replaces the previous blob via parameterized replaceable event semantics ([NIP-33](33.md)).

#### Read-Before-Write

Before publishing, a client MUST:

1. Fetch its own current blob (identified by decrypting and matching `client_id`) from each relay it intends to publish to, and merge all fetched versions.
2. Merge the fetched blob with local state using `max()` per context.
3. Publish the merged result.

The read-before-write rule applies to all publishes, including convergence re-publishes triggered by incoming blobs from other client instances.

The `created_at` monotonicity rule applies relative to the maximum `created_at` seen across all fetched blobs for the same `d` tag. This ensures no state is lost when two instances write concurrently.

#### Live Subscription and Convergence

Clients SHOULD maintain a live subscription to all `kind:30078` events for their pubkey with `#t: ["read-state"]`. When a blob from another client instance arrives:

1. Decrypt it and merge into local state using `max()` per context.
2. If any context timestamp in the incoming blob is greater than the corresponding timestamp in the client's last-published blob (or the context is absent from the last-published blob), re-publish the client's own blob after a debounce delay.
3. Clients MUST suppress the re-publish if the merged result is identical to the client's last-published blob. A client that has never published treats its last-published blob as empty.
4. Clients SHOULD limit re-publishes triggered by incoming blobs to at most one per debounce window, regardless of how many blobs arrive during that window.

This drives convergence without a coordination round-trip. Each client only updates its own blob based on what it specifically reads — it does not republish other clients' data.

#### Clock Skew

When publishing, if the client's local clock produces a `created_at` value less than or equal to the maximum `created_at` seen across all fetched blobs for the same `d` tag, the client MUST use `max_fetched_created_at + 1` instead.

#### Debounce and Pruning

Clients SHOULD debounce writes to avoid excessive relay traffic (e.g., flush 5–10 seconds after the last local read-state change, or on app close/background transition). Clients MUST NOT write on every individual read action.

The blob SHOULD contain only contexts the client has explicitly interacted with. Clients SHOULD prune aggressively, prioritizing recently-active contexts, and MAY drop entries older than the time horizon before writing. Clients MUST ensure the published event does not exceed relay event size limits (typically 64 KB content).

#### Client ID Rotation

Clients MAY rotate their `client_id` (e.g., after a reinstall or for privacy reasons). Rotation creates a new blob under a new slot ID; the old blob becomes orphaned and ages out of the time horizon naturally. This does not affect reading — the merge rule still incorporates the old blob until it expires.

#### Orphaned Blob Deletion

When a client rotates its `client_id` or is decommissioned, the old blob becomes orphaned. Clients MAY publish a `kind:5` deletion event ([NIP-09](09.md)) referencing the old event to request relay-side removal. This is a courtesy to reduce storage — relays are not required to honor deletion requests, and the blob will age out of the time horizon regardless.

## Example

A user runs two clients: a desktop app (`client_id: a1b2c3d4`, slot ID: `7f3a...`) and a mobile app (`client_id: e5f6a7b8`, slot ID: `9c0d...`).

Desktop blob (`d` tag: `read-state:7f3a...`), decrypted content:
```json
{
  "v": 1,
  "client_id": "a1b2c3d4",
  "contexts": {
    "ctx:AAA": 1700000100,
    "ctx:BBB": 1700000050
  }
}
```

Mobile blob (`d` tag: `read-state:9c0d...`), decrypted content:
```json
{
  "v": 1,
  "client_id": "e5f6a7b8",
  "contexts": {
    "ctx:AAA": 1700000200,
    "ctx:CCC": 1700000080
  }
}
```

Merged effective state (computed by either client):
```json
{
  "ctx:AAA": 1700000200,
  "ctx:BBB": 1700000050,
  "ctx:CCC": 1700000080
}
```

The relay operator sees two events with `d` tags `read-state:7f3a...` and `read-state:9c0d...` — both opaque. They cannot determine which device produced which blob.

## Test Vectors

The following test vectors use the decrypted plaintext form. NIP-44 encryption/decryption is covered by NIP-44's own test vectors and is not duplicated here.

### Single blob, single context

Input blobs (decrypted):
```json
[{"v": 1, "client_id": "aaa", "contexts": {"ctx:X": 1700000100}}]
```

Expected effective state:
```json
{"ctx:X": 1700000100}
```

### Two blobs, overlapping contexts (max-wins)

Input blobs (decrypted):
```json
[
  {"v": 1, "client_id": "aaa", "contexts": {"ctx:X": 1700000100, "ctx:Y": 1700000300}},
  {"v": 1, "client_id": "bbb", "contexts": {"ctx:X": 1700000200, "ctx:Z": 1700000050}}
]
```

Expected effective state:
```json
{"ctx:X": 1700000200, "ctx:Y": 1700000300, "ctx:Z": 1700000050}
```

### Unknown schema version ignored

Input blobs (decrypted):
```json
[
  {"v": 1, "client_id": "aaa", "contexts": {"ctx:X": 1700000100}},
  {"v": 99, "client_id": "bbb", "contexts": {"ctx:X": 1700000999}}
]
```

Expected effective state (v:99 blob ignored entirely):
```json
{"ctx:X": 1700000100}
```

### Empty contexts

Input blobs (decrypted):
```json
[{"v": 1, "client_id": "aaa", "contexts": {}}]
```

Expected effective state:
```json
{}
```

## Privacy Considerations

The `content` field is NIP-44 encrypted to the user's own keypair. Context identifiers, timestamps, and client identifiers are not visible to relay operators or other users. As with all NIP-44 encrypt-to-self data, compromise of the user's private key exposes all stored read state.

The `d` tag suffix and `t` tag are visible to relay operators. They reveal that the user employs read-state sync and the number of active blobs (which approximates the number of client instances). However, the `d` tag suffix is a random opaque string — relay operators cannot link it to a specific device, client implementation, or session. Write frequency may reveal approximate activity level.

The encrypted blob size is proportional to the number of tracked contexts. Relay operators can observe ciphertext length, which leaks an approximate context count. Clients concerned about this MAY pad the plaintext before encryption, though this NIP does not prescribe a padding scheme.

Because the merge rule is monotonic, replaying an old event to a relay is harmless — it cannot lower a read timestamp.

Clients supporting multiple Nostr identities SHOULD use distinct `client_id` values and distinct slot IDs per identity. Reusing either across pubkeys allows relay operators to link those identities.

This feature is opt-in. Clients MUST NOT publish read state events unless the user has explicitly enabled the feature.

| Kind | Usage |
|------|-------|

## Backwards Compatibility

This NIP introduces no changes to existing event kinds or relay behavior. It uses only standard NIP-01 event storage, NIP-33 addressable event semantics, NIP-44 encryption, and NIP-78 application data conventions. Clients that do not implement this NIP are unaffected.
